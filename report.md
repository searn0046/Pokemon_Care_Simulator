# Report

## Intro

This report describes my implementation of the _Pokémon Care Simulator_ (PokGotchi), and how AI tools were used as support during the process.

## Design choices

### Domain model (PokGotchi)
The domain model is organized with a clear separation between simulation logic and user interface. Sprite behavior is implemented in _classes.py_, while event handling, rendering, and widget updates are implemented in _main.py_. This structure was chosen to keep domain rules independent of UI code, so domain logic can be modified without changing interface logic.

At class level, the hierarchy is as follows: _pygame.sprite.Sprite_ -> _Pokemon_ -> type classes (_FireType_, _WaterType_, _ElectricType_, _RockType_) -> concrete species classes (_Charmander_, _Growlithe_, _Vulpix_, etc.). Inheritance was chosen instead of composition because the species classes are stable specializations of a shared Pokémon concept. Each species has an "is-a" relationship to both its type and the _Pokemon_ base behavior, making inheritance a direct representation of the domain.

The _Pokemon_ base class has behavior that should be identical for all species: lifecycle state (*is_alive*, *is_awake*), age tracking, mood calculation, need updates (_nutrition_, _energy_, _happiness_), and shared actions such as _feed()_, _play()_, _sleep()_, and *give_energy()*. Type classes introduce specific (but semi-random) decay rates and energy sources. Species classes are intentionally minimal and primarily define species-specific attributes (images, image scaling factors, and name). This greatly reduces duplicated logic and keeps species definitions easy to extend.

Encapsulation is used to protect the internal state and make updates explicit. Need values are stored as private attributes and exposed through read-only properties, while mutation is handled by methods inside the class. This design prevents external modification of gameplay state, and keeps validation and clamping rules in one place.

Global registries (_pokemon_names_, _pokemon_group_, _pokemon_dict_) are used for uniqueness checks, update iteration, rendering, and UI lookup. This choice prioritizes simple access for a small interactive application, while still keeping object behavior encapsulated inside each Pokémon instance.

### Extension (PokGotchi2)
From the very beginning, I envisioned _Evolution_ as my extension. The starting Pokémon were chosen partly based on their ability to evolve, and I initially created classes for all 32 species, while keeping only base-stage Pokémon in the domain model. This is why the _give energy_ feature was implemented, in addition to _sleep_.

Implementing species evolution also means adding more methods to _Pokemon_, and a system to decide when a Pokémon can evolve. Similar to _nutrition_, _happiness_, and _energy_, the Pokémon objects now have a _level_ attribute as well. A non-final-stage Pokémon levels up when given its respective energy source (provided its energy is also high enough), and evolves after level 10.
As a small bonus, there is the option to spawn a Pokémon of any stage if *ONLY_FIRST_STAGE* is set to _False_. If set to _True_, only first-stage Pokémon may spawn.

To better adhere to the _abc.ABC_ implementation requirement, I changed my previous approach (marking constructors as abstract methods). The _Pokemon_ class now has an abstract *set_images()* method, which is implemented in each species class and called directly by their constructors. The subclasses of _Pokemon_ are therefore responsible for loading their own images. I consider this technically unnecessary for this project, but I believe it demonstrates familiarity with _ABC_ and satisfies the requirement. My impression is that abstract base classes are especially useful in larger projects and in collaborative development.

## AI reflection

### AI level
Having the option to use AI at UiT's level 4 naturally raised the question: "What is the limit?" On the other hand, my personal preference is to first see how much I can manage on my own. I experimented with moving a Pokémon sprite around in the _pygame_ window before consulting with GitHub Copilot in the LLM chat.
Still, the AI use in this project falls under UiT's level 4, seeing as I used AI models to generate both code and images.

### AI tools
####  VS Code -> __GitHub Copilot__
<u>Time-consuming "brainless" work</u>
_Next Edit Suggestions_ was the first AI tool used for this project. VS Code already has useful shortcuts for multi-cursor editing, but GitHub Copilot's _Next Edit Suggestions_ still saved me tremendous amounts of time, especially when editing many subclasses with similar (but not identical) changes. One example was adding image paths and attribute names to all 32 species classes. It also helped catch obvious typos __before__ they crashed my program. The _README.md_ file and UML class diagrams were also generated by GitHub Copilot, but under my supervision.

<u>Tutoring</u>
I explicitly instructed the LLM to act as a tutor, log the tutoring, and not generate code unless I asked for it. There were, however, times when I prompted Copilot to edit code directly (e.g., copying theming parameters from one UI element to another), which made it "break character." In those cases, I had to remind it to stay in "tutor mode."

#### Google Gemini -> __Nano Banana 2__, __Windows Photos__
I used _Nano Banana 2_ via _Google Gemini_ both for altering images (mostly style) and for generating new ones. At the time I worked on this project, _Nano Banana_ was unable to create _.png_ images with transparent backgrounds (despite "trying"), so a significant amount of time went into removing backgrounds, cropping images, and adjusting brightness and saturation. This was done using the _Photos_ app on Windows. I specifically mention the _Photos_ app because its background-removal feature is AI-driven.

### Progressive interactions
GitHub Copilot gave useful advice on UI element choices, e.g. using a _UIPanel_ as a container with separate _UIButton_ elements for the Pokémon selection menu instead of a _UISelectionList_. Since I wanted each active Pokémon's name __and image__ in the selection widgets, this advice likely kept me from spending unnecessary time on something impractical/impossible.

### Difficulties
_Next Edit Suggestions_ can definitely save time, but the option to disable it is absolutely __vital__. When I have an idea, start typing, and the agent immediately wants to generate an entire code block for me, that is when my ego kicks in. I do not want my potentially brilliant idea to come off as __predictable__.

### Impacts on learning process
When learning and improving any skill, you need to struggle a bit. As a rule of thumb, more struggle leads to more effective learning. But if one remains stuck for too long and starts learning that "_you won't manage_" or "_you'll have to wait for help_," then having immediate tutoring from an LLM is very useful. Much like Pokémon creatures, humans also have limited energy.

### AI use for software development
I would recommend keeping AI use proportional to one's own level. The work the AI does for you should be something you could have definitely done yourself with more time. When the inevitable need to change or debug something arises, it is undoubtedly better to know what is happening in each part of the code. Software development is a career that requires continuous learning, and to compete in the job market, one must go beyond simply prompting an LLM.
Beyond that, I would advise consistently prompting the LLM to check documentation before answering queries. This helps prevent the model from leading you astray with guesses based on outdated information.

### Future mandatory assignments
"Googling it" has somewhat changed its meaning. For future assignments, I will absolutely want AI models to generate examples of syntax and overviews of functionality in Python library documentation, and in Python itself.
To the extent allowed, I will also use AI tools to reduce repetitive workload so that I can focus on work that requires more active thinking.

## Misc.

### The best part
I am very happy that this assignment was without pre-code. Building something from the ground up gives more freedom to decide what to do and how to do it. It also gives a stronger sense of ownership of the project. This aspect, combined with the option to use AI for help, has been quite motivating.

### Challenge
One big challenge for me has been deciding when it is time to move on. I often get stuck on an imperfection, spending excessive time improving something that is not even required.
Another challenging and time-consuming aspect has been GUI theming. What should be defined in the Python script, and what should be defined in _theme.json_? Debugging _theme.json_ would be much harder without AI assistance, as common issues such as invalid keyword parameters often produce no useful error or warning messages.

### Developer practices
Using virtual environments for dependency/version control is something I was already familiar with from previous online courses. This is essential in Python, since most projects rely heavily on third-party modules, and these modules can change significantly between versions.

But I __have__ learned a lot from this project. Even if you know the basic theory of class structures and game loops, nothing is more educational than trial and error. I am now more proficient at recognizing different error types, and debug more efficiently than before.
Git is also a new tool in my toolbox. Running commands that change, remove, or add local files can be a bit frightening at first, but there are clear benefits to using it.
I am yet to adopt a more structured workflow. For my next assignment, I consider planning my project with UML diagrams and a time budget for each step, and perhaps using _Pytest_ as part of the process.
